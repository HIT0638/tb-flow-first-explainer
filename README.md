# tb-flow-first-explainer

面向 AI 编程助手的"流式优先解释"技能，为 tb 的学习风格定制。

## 核心思想

**先用眼睛看，再跟着走，最后才命名。**

不从"为了实现 X，系统引入了 Y"开始，而是从一个具体、可见的对象出发，跟踪它如何移动、触碰谁、依赖什么、在哪里可能断裂——然后在链条清晰之后，再总结设计目的和术语。

## 回答结构

1. **具体角色**：它是什么？直接管理、接收、暴露、翻译、存储、调度、执行什么？
2. **最小工作单元**：它直接触碰的最小对象是什么（记录、任务、文件、消息、分区……）？
3. **外与内**：外部暴露的名字/抽象是什么？内部真正干活的是什么？
4. **一条活链**：跟踪一个请求/记录/事件/状态变化从头到尾的移动路径
5. **依赖与锚点**：在这条链上，它从哪里获得地址、状态、身份、归属、协调信息？
6. **断裂点**：如果没有它，链条会在哪里断开、歧义、重复、丢失状态或压力集中？
7. **然后命名目的**：最后才总结设计目标（吞吐、可用、解耦、一致性……）

## 使用方式

将此技能添加到 AI 编码助手（如 opencode）的技能目录中，当 tb 提出"帮我理解某模块/机制/代码路径"类问题时自动触发。

回答应该让人感觉是"跟着这个对象走一遍"，而不是"列出所有组件和属性"。
